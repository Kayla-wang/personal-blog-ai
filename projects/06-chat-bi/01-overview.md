# 概述:为什么 Text2SQL 不够可靠

Chat BI 的愿景很诱人:业务同学用自然语言提问,系统自动生成 SQL、查库、返回结果。可一旦落到真实生产,直接把问题丢给大模型「写 SQL」的方案,可靠性远不够用。这个系列会用 Node.js + TypeScript 复刻 [SuperSonic](https://github.com/tencentmusic/supersonic) 的核心思路,搭一个极简但结构完整的 Chat BI —— mini-SuperSonic,借它讲清「为什么要有语义层」和「智能体循环该加在哪」。

## 一、痛点:裸 Text2SQL 为什么不靠谱

裸 Text2SQL 指的是「NL → 大模型 → SQL → 执行」这条最短流水线。它在 demo 里效果惊艳,在生产里却频繁翻车,原因不在模型不够聪明,而在这条链路把太多决定权交给了模型:

- **表结构理解**:模型要猜哪张表、哪个字段代表「销售额」。
- **业务口径**:模型要猜「活跃用户」是 `COUNT(user_id)` 还是 `COUNT(DISTINCT user_id)`。
- **拼 SQL**:模型要同时保证语法正确、join 正确、聚合正确。

任何一环猜错,结果就错。而且这些错误往往**不报错**。

## 二、最危险的是「静默错误」

传统软件的错误大多会「响」:抛异常、报 500、编译不过。Chat BI 最危险的错误却是**静默**的 —— 语法完全正确、SQL 顺利执行、返回一个看起来很合理的数字,但它是错的。

举个例子。用户问「上个月各地区的活跃用户数」,模型生成:

```sql
SELECT region, COUNT(user_id) AS active_users
FROM orders WHERE month = '2026-06'
GROUP BY region;
```

看着没问题,但「活跃用户」的正确口径是**去重**计数 `COUNT(DISTINCT user_id)`。同一个用户当月下了 3 单,这里就把他数了 3 次。查询成功、有结果、数字偏大 —— 没有任何信号告诉你它错了。业务同学拿着这个膨胀的数字去做决策,才是真正的风险。

## 三、一个判断框架:验证信号 × 犯错代价

判断一个 AI 应用「能不能放心用」,可以看两个维度:

| 维度 | 含义 | Chat BI 的处境 |
| --- | --- | --- |
| 验证信号 | 结果对错,系统能否自动判定 | 弱:SQL 能跑通不代表口径对 |
| 犯错代价 | 错了之后损失多大 | 分场景:只读查询本身无副作用,但错误结论会误导决策 |

Chat BI 的尴尬正在于此:它**只读**,所以不会像「AI 自动改数据库」那样一步惹祸,落地门槛低;但它的**逻辑对错缺乏自动验证信号**,静默错误能一路传到决策层。低门槛 + 无信号,合起来就是「容易上线、也容易悄悄骗人」。

## 四、解法概览:三层兜底

mini-SuperSonic 不追求让模型「一次写对 SQL」,而是把可靠性拆成三层兜底:

1. **语义层兜住「理解」**:预先定义好实体、维度、指标、口径,模型只需产出结构化的「语义查询」,而不是直接写 SQL。口径由引擎按定义编译,模型碰不到。
2. **评测集造「验证信号」**:逻辑对错本没有自动信号,那就用黄金样本集人为造一个 —— 固定一批问题和期望答案,回归时自动比对。
3. **人兜「结论」**:分步透明地打印每一步中间产物,让使用者能看懂系统「怎么理解的」,在结论层做最后把关。

## 五、本系列要做什么

mini-SuperSonic 的核心是一条五步链路,外加一个纠错循环:

```
用户问题(NL)
   │
   ▼
[1] Schema Mapper   规则:把词映射到指标/维度/维度值
   │
   ▼
[2] Semantic Parser  LLM + zod:产出结构化语义查询
   │
   ▼
[3] Semantic Corrector  规则校验:口径/取值是否合法  ──校验失败──┐
   │                                                          │
   │ 校验通过                                    把错误喂回 Parser 重试
   ▼                                                          │
[4] Semantic Translator  纯代码:语义查询编译成 SQL   ◀────────┘
   │
   ▼
[5] Executor  执行 SQL,返回结果
```

关键设计:**确定性的部分用代码,不确定的部分才用 LLM**。Mapper、Translator、Executor 都是纯规则/纯代码;只有 Parser 用大模型;而唯一需要「循环」的地方,是 Corrector 校验失败后把错误喂回 Parser 重试 —— 这才是真正意义上的「智能体」。

后续四篇分别展开:

- **02 语义建模**:怎么给数据写一本「业务词典」,以及可加性这个核心概念。
- **03 链路拆解**:一句话到 SQL 的五步旅程,逐组件看代码。
- **04 智能体循环**:Corrector→Parser 的自动纠错怎么搭,何时该加循环。
- **05 验证与总结**:用评测集抓静默错误,以及这个简化版的边界。
